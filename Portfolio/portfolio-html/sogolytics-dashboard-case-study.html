01 Project Overview

The Problem Behind the Problem

Sogolytics is a B2B SaaS Experience Management platform serving clients across three package tiers — from SMB through to enterprise. The platform excelled at survey data collection and reporting. But the analysis layer was failing both the people who build dashboards and the people who consume them.

Account Managers — the internal power users who configure dashboards on behalf of enterprise clients — had no canvas-based builder. Every insight lived in a separate static report, manually exported and emailed. Every configuration change required rebuilding from scratch.

Core Problem

The platform lacked a configurable analysis layer. Account Managers couldn't combine ratings, response tables, score summaries, and contextual labels into a single shared surface — creating a critical churn risk at the enterprise tier against competitors who had already moved to configurable dashboards.

Two Personas. One Critical Handoff.

🏗
Account Manager
Sogolytics Internal · Builder

Configures dashboards on behalf of enterprise clients. Primary usability test participant. Their workflow is the product — if setup breaks down, the client never sees value.

📊
CX Manager / Marketing / PM
Client-side · Consumer

Reads, filters, and shares dashboards to drive decisions. Receives the dashboard the AM builds. Their trust in the data depends entirely on the AM's ability to surface the right signals.

04 Research & Usability Testing

The Moment That Defined the Entire Design

In-House Usability Test

7
Account Managers tested
Internal proxy users — builder persona
100%
failure rate at drag-and-drop
Before design intervention
Primary Finding

Every participant got stuck in the same place — dragging a widget from the left menu and not knowing where it would land on the canvas. The canvas was silent when it should have been speaking.

What the test revealed:

Participants weren't confused about what the dashboard could do — they were confused about where things would land. The gap between intent and feedback was the entire design problem, expressed in one repeated gesture across all 7 sessions.

Secondary friction: widget type affordances (Graph vs Score vs Table) weren't visually distinct enough in the selection menu. AMs had to trial-and-error the right widget for each KPI type.

Completion rate delta — X/7 → Y/7 post-redesign (withheld under NDA)

Design Responses — One Observation, Three Fixes

👻

Ghost Preview

Live visual representation of the widget during drag — shows exactly where it will land before the user releases. Closed the loop between "I'm moving this" and "it will go here."

🎯

Visible Drop Zones

Canvas zones illuminate on drag initiation — making valid placement areas explicit. Users no longer had to guess which regions accepted widget drops.

⚡

Widget Preview Thumbnails

Added labelled visual previews to the left menu widget selection. AMs recognise what they need (NPS chart, response table) rather than having to remember a widget name and guess its output.

Outcome

Task completion improved post-redesign. The fix wasn't a tooltip or onboarding modal — it was a redesigned affordance. The guessing stopped.

05 Design System & Interaction

Four Widget Types. One Complete Information Hierarchy.

The widget taxonomy wasn't arbitrary — it maps directly to how enterprise stakeholders consume data: headline first, trend second, detail on demand. AMs arrange widgets to match this reading order for each client's specific priorities.

📊
Graph Widget

Visualises NPS, CSAT, or any rating question. Shows patterns and distributions. Trend layer.

🔢
Score Widget

Summarises project performance as a single metric. The headline number that leads a dashboard. KPI layer.

📋
Table Widget

Sortable response-level data. Enables drill-down from aggregate to individual. Detail layer.

📝
Description Widget

Adds text, headers, and context. Lets AMs annotate client-specific meaning onto data. Context layer.

Information Hierarchy Design Principle

Description sets context → Score leads with the headline KPI → Graph shows pattern and trend → Table enables individual-response drill-down. Four widgets, four layers of progressive depth.

06 Psychology & Behaviour

Every Interaction Decision Mapped Back to One Principle

Core Design Principle

Enterprise users making data-driven decisions under time pressure cannot afford to find insights — the dashboard has to surface them. Every cognitive load reduction choice served this principle.

Principle
Application in Sogolytics Dashboard
Progressive Disclosure
Step flow: Access → Create → Add Widget → Configure → Customise → Save. Advanced settings only surface after a widget is placed — complexity is earned, not front-loaded.
Immediate Feedback Core Fix
Ghost preview during drag closes the loop between "I'm moving this" and "it will land here." This single principle resolved the failure that affected 100% of usability test participants.
Recognition Over Recall
Widget library uses labelled names + visual previews — AMs recognise what they need rather than remembering a system widget name and guessing its output shape.
Segmentation Flexibility
Widget-level vs dashboard-level segmentation gives AMs fine-grained control without forcing them to build duplicate dashboards for different views within the same client account.
Recipient Trust Calibration
Permissions at share time (filter / drill-down on/off) let AMs match data access to stakeholder trust level. A governance behaviour embedded in the sharing UX — not a separate admin screen.
Habit Formation via Push
Recurring email delivery (Daily/Weekly/Monthly) shifts stakeholder behaviour from pull (log in to check) to push (data arrives in existing workflow) — increasing adoption without requiring new login behaviour.

RBAC as UX, Not Admin

Role-based access control wasn't a post-launch addition. It was designed as a first-class UX feature — permission-setting appears inline at the point of sharing, visible before the share action fires, not discovered after a recipient gets the wrong access level.

This decision had direct governance consequences for enterprise clients handling sensitive employee or customer survey data — particularly in financial services and healthcare verticals where accidental data exposure creates compliance risk.

10 Reflection & Story

What I'd Do Differently. What I'd Defend.

What I'd Do Differently

🔁

Re-test After the Fix

We observed task completion improved post-redesign, but a formal second-round test with the same 7 AMs would have produced a defensible delta — not a qualitative observation. I'd build that into the timeline next time.

📱

Earlier Mobile Signal

Mobile was deferred to v2 as a deliberate scope decision. Given that some end users consume dashboards on mobile, an early prototype test would have surfaced whether the deferral created adoption friction before launch rather than after.

What I'd Defend

🎯

Fixing the Affordance, Not the Onboarding

When all 7 AMs got stuck in the same place, the instinctive response is to add a tooltip or walkthrough. I redesigned the affordance instead. The interaction should speak for itself — documentation is a symptom of a design that doesn't.

🔒

Inline Permissions at Share

Some PMs wanted to simplify the share flow by putting permissions in account settings. I pushed back — governance errors at enterprise scale are expensive. The friction of seeing permissions before sharing is deliberately lower than the friction of fixing a data exposure after.

The Story in One Paragraph

Seven Account Managers. One task: build a dashboard for a client. Each of them knew the platform well. Each of them got stuck in the same place — dragging a widget from the left menu and not knowing where it would land on the canvas. It wasn't a knowledge problem. It was a feedback problem. The canvas was silent when it should have been speaking. The fix wasn't a tooltip or an onboarding modal. It was a ghost preview — a live signal that said this is where it's going before you let go. Task completion improved. The guessing stopped. A dashboard is only as trustworthy as the experience of building it. Precision at the builder layer produces confidence at the stakeholder layer.
