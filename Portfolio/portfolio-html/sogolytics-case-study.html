01 — RESEARCH

Understanding the Pain

User interviews were conducted across three core user types — data analysts, marketing managers, and business owners — surfacing a consistent pattern of frustration with filter creation. The research focused on four problem areas: filter setup difficulty, multi-filter management, data analysis efficiency, and insight extraction.

⏱

Time Tax on Filter Creation

Users consistently reported spending more time configuring filters than analyzing results. Complex multi-condition filters required manual reconstruction every session with no save or reuse functionality.

🔁

Redundant Multi-Filter Work

Analysts routinely built multiple overlapping filters to extract a single insight — a direct symptom of no combinatorial logic or filter grouping support in the existing interface.

🧭

No Intelligent Guidance

The platform offered zero contextual suggestions. Users had to recall variable names, data types, and valid ranges from memory — creating high cognitive load and frequent errors.

Key interview insight: "I know what I want to find in the data. The system doesn't help me express that — I have to translate my question into its language, and that translation is where I lose time and make mistakes."

02 — DESIGN DIRECTION

How Might We?

Design Challenge

"How might we transform filter creation from a cognitive burden into an intelligent, guided experience that anticipates what analysts need before they ask?"

Principle 01

Simplicity First

Reduce the number of steps required to create a functional filter. Every interaction should feel obvious without documentation.

Principle 02

Intelligence as Default

Smart filters should automatically surface relevant suggestions based on the active dataset — no manual configuration required to get started.

Principle 03

Persistence & Reuse

Filters built once should be saveable, shareable, and discoverable. The system should remember, so the analyst doesn't have to.

Constraint to honor: All improvements must operate within the existing Sogolytics design system and brand guidelines, and remain compatible across SogoSurvey, SogoCore, SogoCX, SogoEX, and SogoConnect.
03 — SYSTEM ARCHITECTURE

Information Architecture

The filter system was positioned as a cross-cutting module — Sogo Filter Manager — that lives above the product layer and feeds into all five products. This centralized IA eliminates the fragmented per-product filter experiences that existed before.

How the New Filter Flow Works

A step-by-step walkthrough of the redesigned filter creation experience.

Contextual Entry Point

Users access the filter interface from any Sogolytics product. The system automatically detects the active dataset and pre-loads relevant field suggestions — no blank canvas to face.

Smart Filter Suggestions

Based on the current dataset and user history, the system surfaces the top 5 most relevant filter starting points. Analysts can accept a suggestion or build from scratch.

Guided Condition Builder

A redesigned condition builder uses inline autocomplete for field names, auto-validates data types, and shows live result counts as conditions are set — preventing empty-result errors before they happen.

Logic Group Controls

AND/OR grouping is surfaced prominently rather than buried. Users can create nested logic groups with drag-and-drop reordering — handling complex multi-condition filters cleanly.

Save, Name & Share

Filters can be saved with custom names and tags, organized into folders, and shared with specific team members or the whole organization — creating a reusable filter library over time.

Filter Analytics Dashboard

Administrators and power users see which filters are most used, identify stale filters for cleanup, and track adoption across the five products from a single analytics panel.

06 — REFLECTION

Lessons & Next Steps

What I Learned

  • Centralizing a cross-product feature requires buy-in from multiple product teams early — stakeholder alignment was as critical as the design work itself.
  • Smart suggestions only feel smart when they're accurate. The filtering logic needed close collaboration with engineering to ensure suggestions were contextually valid, not just syntactically possible.
  • Users don't just want powerful filters — they want to feel confident using them. Inline validation and live result counts dramatically reduced user anxiety during testing.
  • The "blank canvas" problem is real. Starting users with suggestions instead of empty fields increased task completion rates in prototype testing.
  • Filter naming and organization became an unexpected priority — power users had 50+ saved filters and no way to manage them. Folder and tagging system was a late but essential addition.

Next Steps

  • Instrument post-launch analytics to capture baseline metrics for filter creation time, error rate, and suggestion acceptance rate.
  • Expand smart filter intelligence with ML-based suggestions trained on anonymized usage patterns across the platform.
  • Build a filter template library — allowing Sogolytics to ship pre-built "starter filters" for common use cases (e.g., NPS analysis, employee engagement baseline).
  • Investigate natural language filter input — allowing users to type queries like "show responses from Q3 where NPS > 8" and auto-generate the filter logic.
  • Explore mobile-responsive filter management for on-the-go access by executive stakeholders reviewing dashboards on tablets.
