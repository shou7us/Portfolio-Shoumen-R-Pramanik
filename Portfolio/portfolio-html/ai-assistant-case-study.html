01 · Overview

One assistant.
Every module. Zero friction.

The XM Platform helps enterprises collect, analyse, and act on experience data across the customer and employee lifecycle. Powerful — but complex. Non-technical users routinely needed analyst support to filter dashboards, configure surveys, or interpret reports.

The AI Assistant changes this. Embedded across Survey Creation, Survey Distribution, Reports, CX Dashboard, and Custom Dashboard — it lets users type or speak naturally, and the system translates intent into live API calls that execute immediately.

The Problem

Power users could unlock deep insights; everyone else was stuck. High cognitive load, steep learning curves, and analyst bottlenecks meant most platform capability went unused.

The Opportunity

LLM-powered natural language interfaces could close the gap between intent and action — but only if the UX made the AI feel trustworthy, fast, and genuinely useful within each context.

My Role

Led UX strategy, interaction design, and prompt-library architecture. Collaborated with Product, Engineering, and Data Science across all five module integrations from discovery to launch.

Primary Users

CX Managers, HR Business Partners, and Operations teams who use the platform daily but don't have a technical or data background. Decision-makers who need answers fast.

02 · Problem Framing

The gap between
intent and execution

Through stakeholder interviews and session recording analysis, four core friction patterns emerged across the platform. These weren't edge cases — they were the everyday experience of the majority of our users.

🧱

Steep Cognitive Load

Users needed to know which filter, which widget, and which menu path — before they could even ask their question. The tool demanded expertise it hadn't taught.

⏳

Analyst Bottleneck

Non-technical users raised internal tickets for dashboard edits, custom report slices, or survey logic changes — adding days of wait time to routine tasks.

🔀

Context Switching

Insights lived across modules. Getting a complete picture required navigating Reports, CX Dashboard, and Comparison views separately — losing context each time.

🧭

Discoverability Failure

Platform capabilities (tag filtering, comparison views, NPS drill-downs) were often unknown to users. Features existed; they just couldn't find them without training.

Gap → Opportunity 01

No cross-module AI exists in XM

Every competitor's AI is scoped to a single surface — a survey builder, a dashboard, or an analytics query. None connects across the full platform lifecycle. Our assistant's cross-module awareness is an unoccupied position.

→ Became: unified AI context layer across all 5 modules
Gap → Opportunity 02

AI shows insights but never acts

Qualtrics and Medallia surface AI-generated insights as read-only text. They describe what to do — but the user still has to navigate the UI to do it. No competitor closes the loop from insight to executed API action.

→ Became: NL → API translation with preview-and-confirm execution
Gap → Opportunity 03

Blank prompt = abandoned session

Generic LLMs have the conversational UI — but no XM context and no prompt library. Platform-native tools have the context — but no conversational UI. Nobody combines contextual prompts with native data access.

→ Became: module-aware Prompt Library as the primary onboarding mechanism
Watch: Competitive Threat

Qualtrics is moving toward NL

Qualtrics has announced NL-querying features in their analytics layer. The window for establishing NL-to-action as a platform-wide capability — not just analytics — is 12–18 months before enterprise parity pressure arrives.

→ Ship cross-module execution and multimodal input before feature parity catches up

05 · Research & Insights

What users actually
needed to say

We ran contextual interviews with 12 users across CX, HR, and Ops roles, analysed support ticket clusters, and reviewed session recordings. The core finding: users had clear questions — they just didn't know how to translate them into platform actions.

Observation
User Need
Design Implication
Filter Blindness
"I know I want to see detractor scores for APAC Q3 — I just can't find where to filter it."
AI must translate business questions into filter combinations and execute them instantly.
Survey Logic Fear
Users avoided branching logic in Survey Creator because the UI felt like code.
NL input for survey logic should abstract away conditional rules entirely.
Blank Canvas Problem
"I don't know what to ask the AI. I'm not sure what it can do."
A context-aware prompt library is non-negotiable — remove the intimidation of an empty input.
Trust Deficit
Early prototype testers were afraid the AI would "break something" or make irreversible changes.
Show a preview of every action before execution. Always require confirmation for destructive operations.
Multi-module Queries
"Can I compare NPS from last quarter with this quarter's survey responses in one place?"
The assistant needs cross-module awareness — not just per-page context.

07 · Solution Design

The AI assistant,
built from principles up

Three design principles governed every decision: Context over generic (the assistant knows which module you're in), Preview before action (show what will happen before it does), and Always escapable (users can always undo or cancel).

Natural Language → API Pipeline

Every user message passes through a structured interpretation pipeline before any action is taken on the platform.

01
User Input
Text · Speech · File — any modality
02
Context Injection
Current module + active filters + user role
03
Intent Classification
Query / Action / Navigate / Create
04
API Translation
Maps intent to platform API parameters
05
Preview & Confirm
User reviews before any change is made
06
Execution
API call fired, UI updates in real-time

Integrated across all platform modules

The assistant panel is persistent — a collapsible side drawer — available from every module. Context switches automatically when the user changes screens.

Survey Creation — Generates survey questions, logic branches, and scales from plain-English briefs
Survey Distribution — Configures audience segments, scheduling, and channel rules via conversation
Reports — Filters, cross-tabs, and insight summaries triggered by natural-language queries
CX Dashboard — Smart tagging, comparison views, and filter application (as shown in the product)
Custom Dashboard — Builds, edits, and arranges widgets from conversational instructions

08 · Multimodal Input

Three ways to talk
to the platform

A single AI intent layer supports three input modalities — text, speech, and file — each mapped to a distinct user context and need. Users move between them fluidly; the same pipeline interprets all three. No mode is second-class.

⌨️
Mode 01 · Text

Type your intent

The primary input. Supports full natural-language queries, slash-command shortcuts, and in-line prompt chip selection. Works everywhere the assistant panel is open.

Show NPS trend for APAC
filtered by detractors, Q1 2026
  • Auto-complete for platform terminology (metric names, tag names, date ranges)
  • Slash "/" command triggers quick-action menu for power users
  • Inline prompt chips reduce typing to zero for common queries
  • Multi-turn conversation with context retained across follow-up messages
🎙️
Mode 02 · Speech

Speak your query

Voice input transcribed and interpreted in real-time. Built for on-the-go managers reviewing dashboards in meetings, on mobile, or away from a keyboard.

  • Live transcription shown in the input field as the user speaks
  • Filler word and pause filtering before intent classification
  • Disambiguation prompt shown if query is ambiguous — user speaks or types to clarify
  • Voice confirmation supported — say "confirm" or "cancel" for action approval step
📎
Mode 03 · File

Upload for context

Users drag or attach files — CSVs, PDFs, images, or Excel sheets — and the assistant reads, interprets, and acts on them. Bridges external data into the platform workflow.

📄

  • Image upload (e.g. screenshot of competitor survey) → AI drafts a matching survey structure

  • Drag-and-drop into any module — file type auto-detected, intent inferred from context

09 · Prompt Library

Curated starting points
for every context

Research showed users froze at a blank AI input. The Prompt Library solves this: a curated set of module-specific, role-aware suggestions that appear before the user types — and adapt as soon as they do. It lowers the barrier to first use and teaches users what the AI can do by showing rather than telling.

10 · Key Design Decisions

The choices that
defined the experience

Every major decision involved trade-offs between speed, safety, and learnability. Here are the three that most shaped the final product.

Assistant UI Pattern

✓
Persistent side drawerStays in context without disrupting the current view. Collapses to icon. Always accessible.
–
Floating bubble / chatbotFelt disconnected from page content. Users missed the link between query and result.
–
Full-page AI modeContext loss when switching back to dashboard. Felt like leaving the product.

Action Confirmation Model

✓
Preview + single confirmShow a diff of what will change before execution. One click to approve. Builds trust without adding friction.
–
Auto-execute all actionsFaster, but early testing showed high anxiety — especially for survey publication and distribution changes.
–
Multi-step confirm wizardToo slow. Users abandoned mid-flow. Defeat the purpose of "instant" AI interaction.

Prompt Library Trigger

✓
Always-visible chipsShown before typing begins. Updates contextually. Acts as education layer — teaches capability by showing examples.
–
Triggered on "/" commandPower users loved it; general users never discovered it. Adoption gap too large.
–
None (blank input only)Research confirmed blank canvas paralysis. Drop-off rate in prototype was 68% before first AI interaction.

12 · Reflection

What I learned
building this

What Worked

  • Persistent side drawer kept AI contextually grounded — the right call over a chatbot popup
  • Module-aware Prompt Library solved blank-canvas paralysis and became the primary adoption driver
  • Preview-before-execute built the trust baseline needed for users to take AI-initiated actions seriously
  • Early Engineering collaboration ensured the NL → API translation was scoped to what the platform could actually do

What I'd Improve

  • Earlier usability testing on multi-step intent (queries that span more than one module action) would have saved a late-stage redesign
  • The voice input path was de-scoped late — should have been deprioritised from the start, not removed after design was complete
  • Error states for failed API translations were underdesigned in v1 — users were left uncertain when the AI couldn't interpret a query

Next Steps

  • Proactive AI: assistant surfaces insights unprompted when anomalies are detected in live data
  • Conversation memory: retain context across sessions so users don't re-explain their role and preferences each time
  • Cross-module orchestration: single query triggers coordinated actions across Reports and Dashboard simultaneously
  • Voice-first mode for mobile app variant, targeted at on-the-go CX managers
