01 — Project Overview

The Problem Behind the Problem

Sogolytics is a B2B experience management platform serving data analysts, marketing managers, and business owners across products spanning survey creation (SogoSurvey), CX analytics (SogoCX), employee experience (SogoEX), core data management (SogoCore), and integrations (SogoConnect). At the time of this project, the platform was undergoing a significant strategic shift — from a feedback tool to an enterprise-grade experience management suite.

That strategic shift exposed a critical gap: every product had its own filter system, built independently, with no shared logic, interaction model, or management layer. A data analyst moving between SogoSurvey response analysis and SogoCX customer insights encountered entirely different filter behaviours — different UI patterns, different filter persistence rules, and no way to reuse or share filters across surfaces.

Core Problem

Power users who relied on filters for data analysis had to re-learn filtering every time they changed products, manually recreate filters that logically should transfer, and manage growing filter libraries with no organisational infrastructure — creating avoidable cognitive load and analysis friction at exactly the moment users needed clarity most.

The project mandate was to redesign the filter creation experience across all products and introduce smart filters. I reframed that mandate: the real problem wasn't filter UI — it was the absence of a unified filter architecture. Designing screens before solving the system would have produced five marginally improved isolated experiences, not a platform-ready solution.

02 — History

How We Got Here — And What We Refused to Repeat

Origin State
Siloed Product Development
Each Sogolytics product was built independently, with filter functionality designed by individual product teams. The result was five filter systems with no shared interaction model, logic layer, or persistence strategy.
User Friction Surfaces
Power Users Hit the Ceiling
As usage matured, analysts began working across multiple products. The cognitive overhead of re-learning and re-creating filters became a documented pain point — surfaced in user interviews and journey mapping as a top frustration at the analysis phase.
Strategic Context
Platform Rebranding Exposed the Gap
The company's shift toward enterprise experience management positioned Sogolytics as a multi-product suite. Enterprise buyers evaluate platforms holistically. Fragmented filter UX directly undermined the "unified platform" narrative.
This Project
Architecture-First Redesign
Rather than improving each product's filter UI in isolation, I advocated for and led a cross-product filter architecture redesign — establishing shared logic, a unified Filter Manager, and smart filter suggestions as a platform-level capability.
What We Refused to Repeat

"We won't design another great filter for a single product. We will design one filter system that every product inherits — or we will have this conversation again in 12 months, with more products and more debt."


Lessons from Past Approaches

Previous iterations had treated filter improvements as feature-level work — adding more filter options, improving individual filter UIs, or expanding filter criteria. Each improvement was locally correct and systemically insufficient.

The pattern was clear: filter improvements that ignored cross-product architecture created local wins and platform debt simultaneously. A user who learned a new filter UI in SogoSurvey had to unlearn it in SogoCX. Documentation couldn't fix an architecture problem. Onboarding flows couldn't either.

The Strategic Reframe

The mandate was "improve filter creation and introduce smart filters." My reframe to the PM: smart filters built on top of fragmented architecture would smart-suggest into five inconsistent systems. The intelligence layer depended on a consistent data model underneath it. Architecture was not a nice-to-have prerequisite — it was the product.

03 — Geography & Demographics

Who Uses This, and Where

Sogolytics is headquartered in Virginia, USA, and serves a predominantly North American B2B customer base, with enterprise clients spanning marketing agencies, financial services, healthcare organisations, and retail. The primary user profile for this project was data professionals operating in multi-product enterprise environments — users whose data analysis workflows crossed product boundaries regularly.

While the platform serves global clients, the core filter use case — cross-product analysis by a data analyst or marketing manager — is culturally consistent: analytical users across geographies share the same mental model of filter logic. The design challenge was not cultural; it was structural. A filter system that works for a marketing analyst in Chicago should work identically for one in London or Sydney, because the cognitive model of filtering data doesn't change by geography.

Inclusivity Approach

The filter architecture was designed to be language-agnostic at the interaction level — filter logic (AND/OR operators, criteria fields, saved states) should function identically regardless of locale. Smart filter suggestions were scoped to English-language label matching in v1, with i18n infrastructure flagged for roadmap.

04 — Civics & Regulations

Compliance Shaped the Filter System — Not Just the Visuals

Filter design in an enterprise B2B context carries regulatory weight that goes beyond accessibility. When filters surface personally identifiable data, segment employee populations, or expose customer records, the design of filter sharing and permissions is a compliance surface — not just a UX surface.

Inline permission controls at share action — not buried in settings. Explicit scope selection before filter is shared externally.
Standard
RBAC (Role-Based Access)
Enterprise clients required filter access to be governed by user role. An account manager should not be able to share filters that expose data outside their permission scope.
Scope
Filter Manager & sharing
Filter library visibility, sharing permissions, team/individual access controls.
Design Impact
RBAC surfaced as a UX element — not admin configuration. Users see permission scope before sharing, not after a governance error.
RBAC as UX, Not Admin

The PM's initial proposal was to handle filter permissions in account settings. I pushed back: at enterprise scale, governance errors in filter sharing can expose individual-level survey responses or employee data to the wrong audience — a potential compliance incident for the client. The friction of seeing permissions before sharing is lower than the cost of correcting a data exposure after. Permissions stayed inline at the share action.

06 — Physics & Mechanics

How the Unified Filter System Works

The core design question was not "what should the filter UI look like?" — it was "what is the correct mental model for filtering across a multi-product platform, and how does every product surface inherit from that model?" The following is how the system was designed to function.

Filter Architecture — System Walkthrough
1️⃣

Universal Filter Logic Layer

A shared filter logic model defined at the platform level — not per-product. Criteria types (date range, demographic, response value, segment), operators (AND/OR/NOT), and nesting rules are standardised. Every product consumes the same model.

2️⃣

Product-Contextual Criteria

While the logic layer is shared, each product surfaces context-relevant criteria. SogoCX exposes CX-native fields (NPS score, CSAT band, touchpoint). SogoEX exposes employee-specific fields (tenure, department, engagement score). The engine is the same; the vocabulary adapts.

3️⃣

Smart Filter Suggestions

The intelligent suggestion layer analyses dataset structure and historical filter usage to surface relevant criteria. For a marketing manager opening SogoCX for the first time, the system suggests filters based on the dataset in view — not a blank canvas. Powered by the unified logic layer beneath it.

4️⃣

Filter Manager — The Library Layer

A dedicated cross-product filter management surface. Users can organise filters into folders, tag them for retrieval, search across their library, archive unused filters, and share with team members. The library is product-aware — filters are tagged to their origin product but accessible platform-wide.

5️⃣

Governed Sharing

Inline permission controls at the share action. When sharing a filter, the user selects scope (view only / edit / team library) before confirming. RBAC constraints are enforced at this surface — users cannot share filters that expose data outside their permission boundary.

6️⃣

Cross-Product Transfer

Filters created in one product can be transferred to compatible products. A date-range + segment filter built in SogoSurvey can be applied in SogoCore with one action. Incompatible criteria are surfaced clearly — not silently dropped — so users understand what transferred and what didn't.

Interaction Design — Critical Decisions
InteractionDesign DecisionRationale
AND/OR Logic Builder Core Fix Visual logic tree with live preview of result set scope User interviews revealed that multi-criteria filters frequently returned unexpected results because users couldn't visualise what AND vs OR would do to their dataset. A silent text form was replaced with a visual logic tree — showing users how the filter scope changes as they add criteria.
Smart Filter Suggestions Contextual suggestions based on dataset structure, not generic templates Generic filter templates added noise. Suggestions surfaced by analysing the active dataset's available fields — so a user viewing an NPS dataset sees NPS-relevant suggestions, not a list of every possible filter type.
Filter Library Search Full-text search across filter name, tags, and criteria content Users described existing filter lists as unusable at scale. Folder organisation alone wasn't sufficient — users needed to search across content, not just labels.
Cross-Product Transfer Explicit transfer confirmation with criteria compatibility mapping Silent criteria dropping when transferring between products would undermine user trust. Instead: users see which criteria transferred successfully, which were incompatible (and why), and can choose to proceed or adjust. Transparency over convenience.
Inline Permissions at Share Permission scope visible before share is confirmed See Section 04 — Civics. Governance errors in filter sharing have compliance consequences. The friction of an extra step is lower than the cost of an exposure incident.
Filter State Persistence Active filter state persists on session exit and product switch Users who switch products lose context without persistence. Applied filters are maintained across product navigation so analysts don't lose their working state when moving between surfaces.

Constraints and How They Were Addressed

The project operated within four significant constraints:

🧱

Must work within the existing design system

All new filter components were designed as extensions of existing design system tokens — not new patterns. This constrained some visual possibilities but accelerated engineering implementation and maintained UI consistency across the broader platform.

⏱️

Limited development resources — 6 months

The unified architecture approach was a larger upfront investment than per-product improvements. I built the business case with the PM: the architecture cost was a one-time investment; the alternative (five separate improvements) would accrue ongoing design and engineering debt for every new product added to the platform.

🔗

Integration with existing product infrastructure

The filter logic layer had to be designed with engineering input from the start. Weekly cross-functional sessions with the engineering lead ensured the interaction model was technically feasible within existing data architecture — preventing design decisions that looked clean but couldn't be built.

Tradeoffs We Accepted

The unified architecture approach required deferring mobile filter access entirely. Shipping a solid desktop filter architecture in v1 was more valuable than a stretched version that also attempted mobile — which would have diluted the cross-product logic work. Mobile was explicitly roadmapped, not forgotten.


Tradeoff — Smart Filter Scope

Smart filter suggestions were scoped to label-based matching in v1 — not ML-powered inference. This was an intentional constraint: building ML-based suggestions on top of a newly unified architecture would have introduced too many variables. v1 smart suggestions needed to be explainable and auditable. ML inference was roadmapped for v2 once the architecture had real usage data to train on.


What the Architecture Decision Enabled

By investing in the logic layer first, the design created a foundation that future product teams can build on. Adding a new product to the Sogolytics platform no longer means building a new filter system from scratch — it means inheriting the architecture and adding product-contextual criteria on top of it. The design work compounds over time.

FAANG-Level Framing

At platform scale, every design decision is also a policy for future designers and engineers who haven't joined yet. The filter architecture defined what "a filter" means at Sogolytics — consistently, across every surface. That's the leverage point of systems thinking over screen thinking.

The filter system doesn't live in a product. It lives beneath all of them. Understanding how the architecture connects across the platform — and where the value loops are — was the design work that the screen-level work depended on.

Value Loops
🔄

Reuse Loop

A filter built once in SogoCX is available in SogoSurvey. Usage data from reused filters informs smart suggestions — the system learns which filter combinations are valuable and surfaces them proactively. Reuse creates the training signal for intelligence.

🤝

Collaboration Loop

Team-shared filter libraries reduce duplicated analysis work. When an analyst shares a high-value filter to the team library, it becomes available to marketing managers and business owners — democratising analytical capability without requiring everyone to be an expert filter builder.

🔒

Trust Loop

Governed sharing creates auditable filter provenance. Decision-makers consuming filtered reports can see who built the filter, when, and what criteria it includes. Trust in filtered data compounds — once stakeholders trust a filter source, they act on its outputs faster and with more confidence.

Where the System Was Breaking

Before the redesign, every loop above was broken. Filters couldn't be reused, so no usage signal existed for smart suggestions. Filters couldn't be shared, so collaboration was impossible. Filter provenance was opaque, so trust in filtered outputs was low. The architecture wasn't missing a feature — it was missing the connective tissue that makes a platform behave like a system instead of a collection of tools.

Journey Map Insight

The highest-friction stages were not where I initially expected. Early assumptions pointed to "Filter Setup" as the dominant pain point. Design Psychology Principles Applied
PrincipleApplication in Filter Architecture
Immediate Feedback Core Fix The AND/OR logic builder provides a live preview of filter scope — showing users how many records match their current criteria as they build. Eliminates the guessing game of "will this filter do what I think it will?"
Progressive Disclosure Advanced filter options (nested logic, formula-based criteria) are available but hidden by default. New and occasional users see a simplified interface; power users can expand to full configuration without a separate mode switch.
Recognition Over Recall Smart suggestions surface relevant filter criteria based on dataset context. Users select from intelligently surfaced options rather than recalling field names from memory — reducing cognitive load at the point of highest friction.
Segmentation Flexibility The AND/OR/NOT operator model is exposed visually, not hidden behind text. Users can see and modify the logical relationship between criteria without understanding query syntax.
Habit Formation via Library The Filter Manager is designed to reward consistent filter usage — saved filters, usage history, and team sharing create a virtuous cycle where the more you use the system, the more valuable it becomes.
Trust Through Transparency Cross-product filter transfer surfaces incompatible criteria explicitly. The system never silently drops criteria that can't transfer — it shows the user what happened and why, preserving analytical trust.

12 — Reflection

What I'd Defend, and What I'd Do Differently

What I'd Defend

🏛️

The architecture reframe

The decision to stop and question the brief before executing it was the highest-value design action on this project. A 10-year designer's job is not to execute a brief — it's to validate whether the brief is solving the right problem. The reframe added scope initially, but it removed years of accumulating design debt. I'd make the same call every time.

🔒

Inline permissions at share

The PM's preference for account-settings permissions was understandable from a simplicity standpoint. But in regulated industries — which Sogolytics serves — the cost of a permission error isn't UX friction, it's a compliance incident. Governance decisions belong at the action surface, not in a settings page the user visited three months ago.

🛑

Deferring mobile to v2

Attempting mobile filter access alongside a platform architecture redesign would have stretched both work items past their quality threshold. A stable v1 architecture that works completely on desktop was more valuable than a half-complete architecture that also barely worked on mobile. Scope discipline is a design competency, not a concession.

What I'd Do Differently

🔁

Earlier engineering alignment on the logic layer

I brought engineering into the architecture conversation, but later than ideal. Several interaction decisions in the logic builder had to be revisited when they encountered data model constraints. In retrospect, a 2-hour technical design spike with the engineering lead in week 1 would have saved multiple revision cycles in month 3. Architecture work needs engineering input from the first session.

📊

Established a measurement framework before design began

The absence of hard outcome metrics is a gap I'd close on the next project. I would push to define measurable success criteria — filter re-creation frequency, cross-product transfer usage, filter library retention rates — before the first wireframe. Not because the design needed the metrics to be right, but because those metrics are the business case for the next iteration.

🎓

Pushed for a phased rollout over a full launch

A full-platform filter architecture launch made it difficult to isolate which components were driving behaviour change. Phased rollout by product — SogoCX first, then SogoSurvey — would have provided real usage data that informed subsequent product implementations. Full launch masked what was working.

The Story in One Paragraph

The filter architecture project began as a request to improve some screens and ended as a platform-level systems decision that will outlast any single designer's tenure at the company. The design work that mattered most was never visible in a prototype: it was the reframing conversation with the PM, the business case for architecture investment over screen improvement, and the governance decision that put compliance-critical permissions where users actually encounter them. Every screen-level design decision on this project was downstream of those choices. That's the design work that senior practitioners do — and the work I'll keep doing at whatever scale I'm given.
