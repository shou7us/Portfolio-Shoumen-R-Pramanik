Step 01 · Overview

The problem hiding behind the feature request

The platform's Directory module let business analysts store contact data — names, account numbers, credit scores, loan details — but had no way to compute values from it. Every derived metric (customer tenure, dormancy flags, loan maturity, CLV) required a manual Excel export-calculate-reimport cycle. Data went stale. Analysts lost trust in the platform as a source of truth and reached for spreadsheets at the worst moments.

The initial request was simple: add a formula field type. I reframed it as a larger product opportunity: if analysts don't know what formulas to build, a blank formula builder solves nothing. The real value was an AI layer that could read the schema, infer the domain context, and surface computed fields the analyst didn't know they needed — before they asked.

"The aha moment isn't AI wrote my formula faster. It's AI told me I should track customer dormancy, and I hadn't thought of that."

Step 05 · User journey — B2B dual persona

The analyst journey exposed two distinct friction zones: the moment they realise they need a computed field (but don't know which one), and the moment they try to build it (but don't know the formula syntax). Step 06 · Key insights

Five insights that changed the design

01
Schema reasoning beats formula libraries
Observation
Competitors (Airtable, FormulaBot) suggest formulas from generic libraries — they don't read what fields actually mean semantically
User need
Analysts need suggestions that are specific to their schema and domain — not generic templates they still have to configure
Implication
The AI must infer domain context (banking, CX, healthcare) from field name semantics and map it to domain-appropriate derived fields with real thresholds — not just match keywords
02
RBAC and sensitive fields are first-class design inputs, not edge cases
Observation
These constraints were surfaced late in the process — after the UX flow was designed — and required fundamental rethinking of the suggestion scan, visibility model, and trust architecture
User need
Analysts need permission-aware suggestions — seeing an inaccessible field in the panel and being blocked at save creates frustration and erodes trust
Implication
The suggestion scan must be parameterised by the current user's role at query time — not a single universal scan cached for all users. I advocated for this architecture change with engineering and it was adopted.
03
The abstraction paradox: derived fields hide complexity from viewers but not from creators
Observation
A derived field is designed to abstract formula complexity away from the person viewing it. But RBAC and sensitive field policies require the creator to deeply understand what the formula uses before they can set output visibility.
User need
Creators need full transparency at authoring time; viewers need controlled exposure at consumption time. These are different users with different needs in the same feature.
Implication
Authoring UX must be fully transparent (sensitive flags, role implications, null coverage). Viewing UX is gated by creator-set permissions. The output visibility picker becomes a mandatory step — not an afterthought.
04
Bucketed sensitive data is still sensitive data
Observation
An Income Band field (Low / Medium / High) derived from Annual Income (sensitive) feels safe to analysts — it's aggregated, not raw. But with 3 bands across 1,000 contacts where only 20 score High, the label is precise signal.
User need
Analysts working in regulated verticals need to be warned when their derived field creates a proxy for sensitive data — before they publish it, not after a compliance review.
Implication
Any categorisation or bucketing of a sensitive field must inherit a sensitivity warning on the output. The AI flags this pattern at creation time: "This formula is derived from Annual Income (sensitive). Consider restricting output visibility."
05
The explanation-formula alignment problem is the feature's silent killer
Observation
In early concept testing, analysts read the plain-English formula explanation — not the formula itself. If the explanation drifted even slightly from what the formula computed, analysts made wrong analytical decisions without realising it.
User need
Analysts need to be able to trust the explanation as a reliable proxy for the formula — not just as a label. Trust breaks silently when the two diverge.
Implication
I proposed a secondary validation pass — the explanation is re-derived from the formula (not from the user's original intent) and compared semantically before it surfaces. This shipped as a non-negotiable trust gate in the MVP specification.
Step 07 · Decision framework

Three decisions. The rationale behind each.

Decision 1 — What to ship first: suggestions, builder, or both?

OptionProsConsRationale
A — Both modes simultaneously Full value proposition from day one NL builder risk (hallucinated fields, ambiguous intents) ships before trust is established. Two failure surfaces at launch. —
B — Builder only Familiar mental model (blank input + generate) Blank builder recreates the exact friction we're solving. Schema-blind. High hallucination risk without schema context. —
C — Suggestions MVP, builder V1.5← Chosen Bounded, high-trust interaction. Analyst reviews before saving. Acceptance rate data calibrates the builder before it ships. Delays NL builder. Users who want free-form input must wait. Ship suggestions first. Measure acceptance rate and field name match accuracy. Use real data to calibrate the builder's confidence thresholds in V1.5.

Decision 2 — How to prevent hallucinated field names?

OptionProsConsRationale
A — Warning banner on save Low engineering cost Hallucinated name passes validation; warning is ignorable. Silent corruption reaches production. —
B — Post-save runtime error Standard pattern Analyst discovers error in a report, not at creation. Trust break at worst possible moment. —
C — Token chip enforcement (architectural)← Chosen 100% field name accuracy guaranteed — not a guardrail but a technical constraint. Free-text field entry blocked at the builder level. Higher engineering complexity. Requires field picker UI for every formula input. The highest-severity failure mode is silent data corruption in regulated industries. This is a hard technical gate before any beta release — not a UX decision, an architecture decision I drove with engineering.

Decision 3 — How to handle output visibility for sensitive-input derived fields?

OptionProsConsRationale
A — Block all sensitive field usage in formulas Zero risk of sensitive data proxying Blocks legitimate analytical work. Banking analysts specifically need Age Band, Income Tier, Risk Score. —
B — Allow, no special handling No friction Analysts unknowingly create compliance violations. Sensitive data proxied into widely-visible fields. —
C — Allow with mandatory output visibility picker← Chosen Legitimate work proceeds. Creator retains control. Explicit consent moment recorded for audit. Extra step for analysts using sensitive inputs. The consent moment is the design. Sensitive field usage triggers a mandatory visibility step — not skippable — that forces the creator to explicitly decide who sees the computed result. This satisfies legal and creates an auditable trail without blocking analytical work.
Step 08 · Solution overview

Two AI modes. One trust-first flow.

The solution has two modes that converge at a shared validation and visibility layer. The Suggester and the Builder are not parallel options — the Suggester loads first, always. The Builder is the fallback for when no suggestion fits, or when the analyst knows exactly what they want and wants to build it directly.

1
Add Field → Derived Field type selected
Analyst selects Derived Field from the field type picker. AI schema scan begins immediately in the background — parameterised by the current user's role.
AI: silent role-scoped schema scan
2
Suggestion panel loads — top 6 ranked by domain relevance
Cards display field name, one-line business-language rationale, and a confidence badge (High / Medium). Sensitive-input suggestions show an amber flag. Cards are grouped by category: Identity, Financial, Activity.
AI: domain inference + suggestion ranking + sensitivity tagging
3A
Path A — Analyst picks a suggestion
Formula pre-loads into the builder. Live plain-English explanation appears below the formula. Field references are shown as token chips — clickable, not free-text. Analyst can expand to raw formula editor at any point.
AI: formula pre-population + live explanation generation
3B
Path B — Analyst describes intent in plain English
"Describe what you want to calculate" input. If confidence is below threshold, AI asks one clarifying question before generating. Formula generated with token chip references only — no free-text field names.
AI: NL → formula translation, confidence-gated clarification
4
Validation runs — inline, plain-English, before save
Type compatibility check. Transitive circular reference detection to depth N. Null coverage preview: "This formula will compute for 847 of 1,200 contacts. 353 records are missing [Field Name]." All errors surface inline with plain-English fix suggestions.
AI: explanation-formula alignment check (secondary validation pass)
5
Output visibility picker — mandatory for sensitive-input formulas
Who can see this computed result? Everyone / Specific roles / Creator + Admins only. Sensitive-input formulas cannot skip this step. Selection is recorded for audit. A compliance note is attached to the field record.
6
Field saved — async computation begins across all records
Progress indicator on the column header. UI never blocks. Computation is async — stale values show a spinner, not an error. Field is live and auto-updating from this point forward.
AI does the thinking
Schema scanning and domain inference
Suggestion ranking by relevance and completeness gap
NL → formula translation (field-grounded only)
Explanation generation and alignment validation
Type compatibility and circular reference detection
Sensitive input pattern recognition
Analyst does the deciding
Accept, dismiss, or edit any suggestion
Describe intent in plain language when needed
Review formula explanation before committing
Set output visibility — who can see this derived value
The formula is never hidden. The raw editor is always one click away.
Step 09 · Vibe-coded prototype

We built it. Not just designed it.

To validate the onboarding flow and empty state before engineering scoping, we vibe-coded a working prototype of the feature — covering the first-run experience for analysts who have no derived fields yet, the empty state that greets them, and the entry point into the AI suggestion panel. Building it in code (not just Figma) let us stress-test the interaction timing, the AI scan loading state, and the suggestion panel reveal — moments that don't show up in static mocks.

The prototype directly informed three specification changes: the scan loading state needed a progress signal (analysts assumed the panel had failed if nothing appeared in under 2 seconds), the empty state needed a value-lead headline not a capability description, and the suggestion card reveal needed stagger animation to communicate that cards were ranked, not random.

Step 11 · Trust & failure architecture

What I solved before a line of code was written

I ran a structured failure simulation before any engineering began — identifying the scenarios most likely to break analyst trust and designing mitigations into the specification rather than patching them post-launch. This is the work that made legal and compliance comfortable signing off on the feature.

Top 3 pre-launch risks — solved in specification
1
Hallucinated field names causing silent data errors. An AI-generated formula referencing `Account_Open_Date` when the real field is `Date Account Opened` saves without error and returns null for every record. Analyst discovers corrupted data in a report, not at creation. Mitigation: token chip enforcement at the architecture level — free-text field name entry is blocked. This is a hard gate before beta, not a design guardrail.
2
Explanation-formula mismatch propagating bad analytics. Analysts trust the label, not the formula. If the plain-English explanation drifts even 5% from what the formula computes, wrong data enters reports silently. Mitigation: secondary validation pass — explanation is re-derived from the formula, not from the user's original intent, and compared semantically before it surfaces to the user. Ships with the feature, not after it.
3
Zero-suggestion blank panel on sparse schemas. An analyst opens the suggestion panel with a lean Directory (8 fields, abbreviated names). The panel is empty. First impression: the feature is broken or premium-gated. Mitigation: generic fallback library — Full Name concat, Profile Completeness Score, Has Email, Is Contactable — loads with a clear label: "General suggestions — add more fields for personalised ones." Ships on day one.
Step 12 · Reflection — senior calibration

What I'd do differently

What worked
Reframing the problem before designing the solution
The initial brief was "add a formula field type." I reframed it as "analysts don't know what formulas to build" — which changed the entire product direction. Investing in the capability architecture (schema reasoning over template libraries) and the trust architecture (token chips, secondary validation, output visibility picker) before any UI work meant engineering and legal were aligned from week one, not brought in at the end for sign-off.
What I'd do differently
Surface RBAC and sensitive field constraints in week one
These constraints were discovered mid-process and required fundamental rethinking of the suggestion scan, the trust model, and the visibility architecture. On an enterprise platform with access control, permissions are a first-class design input — not an edge case to handle later. I'd run a permissions audit in the first sprint alongside user research, so the entire UX is designed permission-aware from the first wireframe.
What this opens next
Cross-entity derived fields as the platform differentiator
The V2 roadmap item I'm most interested in is cross-entity derived fields — combining Contact fields with Activity fields (Loan Disbursement, Survey Response) to compute things no competitor offers in this context. An analyst computing Total Loan Exposure per customer across all disbursements — live, in the Directory, powering survey targeting — is the moment the platform becomes an analytical system, not just a contact store.
